Deposit Architecture (Sandbox)

This section is additive and focuses only on deposit design for sandbox implementation. The recommended primary API is Stripe Test Mode because it offers the fastest end-to-end testing path with payment intents, webhook simulation, idempotency support, and unified handling for card plus wallet rails.

Sandbox API Recommendation: Start with Stripe Test Mode as the default provider for deposit execution. Keep Open Banking and Crypto flows as adapters behind the same Deposit API contract.

Sandbox Deposit API Contract

  • POST /api/deposits/create: Create deposit intent and return provider payload (client_secret or redirect URL).
  • GET /api/deposits/status/{depositId}: Return current status for polling UI updates.
  • POST /api/webhooks/stripe: Receive provider events and perform idempotent settlement.
  • POST /api/deposits/retry/{depositId}: Retry a failed or expired sandbox attempt.

Deposit State Model

State Who sets it Transition trigger Notes
initiated Deposit API Deposit request accepted Deposit row created with immutable deposit_id.
pending_provider Deposit Orchestrator Payment intent or redirect created Waiting for provider-side completion.
pending_webhook Provider callback flow User finished provider confirmation UI may still poll until webhook settles.
completed Webhook settlement Verified success event + idempotency pass Wallet credited and transaction audit inserted.
failed Provider or API Provider decline or validation error No wallet credit; retry endpoint available.
expired Scheduler/API Intent timeout exceeded User can start a fresh deposit intent.
reversed Admin/System Dispute, refund, or correction Requires compensating transaction record.

Future Adapters (No Core Redesign Needed)

Open Banking and Crypto can be added as provider adapters behind the same deposit contract. They should emit the same status transitions and settlement events so frontend and ledger behavior remain stable.

  • Open Banking adapter: exchange consent token, monitor transfer completion, emit settlement event.
  • Crypto adapter: generate deposit address, track confirmations, emit settlement once confirmation threshold is met.
  • Both adapters must honor idempotency key and wallet credit invariants.
